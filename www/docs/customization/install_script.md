# Install Scripts

After releasing to GitHub, GitLab, or Gitea, GoReleaser can generate and publish
cross-platform installer scripts (`install.sh` and `install.ps1`) to a repository
you control (via SCM APIs or `repository.git.url`).

The `install_script` section is a single object:

```yaml title=".goreleaser.yaml"
install_script:
  # IDs of artifacts to consider. Empty means all IDs.
  ids:
    - default

  # URL template for download links.
  #
  # Default depends on the SCM:
  #   GitHub: 'https://github.com/<repo_owner>/<repo_name>/releases/download/{{ .Tag }}/{{ .ArtifactName }}'
  #   GitLab: 'https://gitlab.com/<repo_owner>/<repo_name>/-/releases/{{ .Tag }}/downloads/{{ .ArtifactName }}'
  #   Gitea:  'https://gitea.com/<repo_owner>/<repo_name>/releases/download/{{ .Tag }}/{{ .ArtifactName }}'
  #
  # Templates: allowed.
  url_template: "https://github.com/acme/foo/releases/download/{{ .Tag }}/{{ .ArtifactName }}"

  # Directory where generated scripts will be committed in the target
  # repository.
  #
  # Resulting committed paths are:
  #   - <directory>/install.sh
  #   - <directory>/install.ps1
  #
  # Templates: allowed.
  # Default: 'www/static'.
  directory: www/static

  # Default installation directories by operating system.
  #
  # install.sh uses linux/darwin.
  # install.ps1 uses windows.
  #
  # Templates: allowed.
  # Default:
  #   linux: '/usr/local/bin'
  #   darwin: '/usr/local/bin'
  #   windows: '$env:ProgramFiles\\{{ .ProjectName }}'
  install_to:
    linux: /usr/local/bin
    darwin: /usr/local/bin
    windows: "$env:ProgramFiles\\{{ .ProjectName }}"

  # Optional message printed before installation.
  #
  # Templates: allowed.
  message_before: "Installing {{ .ProjectName }}..."

  # Optional message printed after installation.
  #
  # Templates: allowed.
  message_after: "Done!"

  # GOAMD64 to specify which amd64 artifact to use when multiple are available.
  #
  # Default: 'v1'.
  goamd64: v1

  # Commit message used when publishing scripts.
  #
  # Templates: allowed.
  # Default: 'Install script update for {{ .ProjectName }} version {{ .Tag }}'.
  commit_msg_template: "Install script update for {{ .ProjectName }} version {{ .Tag }}"

  # Whether generation/publishing is enabled.
  #
  # Templates: allowed.
  # Default: true.
  # Set to false to turn it off.
  enable: true

{% include-markdown "../includes/repository.md" comments=false start='---\n\n' %}
```

## Runtime behavior

Generated scripts:

- are always named `install.sh` and `install.ps1`;
- are committed to `<directory>/install.sh` and `<directory>/install.ps1`;
- default to the current release tag/version (the one used to generate them);
- use `install_to.linux`/`install_to.darwin` (`install.sh`) and
  `install_to.windows` (`install.ps1`) as default installation destinations;
- support `--version` to override which release to install (`install.sh`);
- support `--install-to` to override destination directory (`install.sh`);
- support equivalent `-Version` and `-InstallTo` parameters in PowerShell
  (`install.ps1`).

The scripts always include a mandatory `generated by goreleaser.com` marker, and
generation fails if that marker is missing.

## Artifact precedence

For each supported platform, installers choose artifacts using this order:

1. `UploadableBinary` (`archives.format: binary`);
2. `UploadableArchive` with `tar.gz`/`tgz`;
3. other `UploadableArchive` formats.

If multiple candidates exist for the same platform, GoReleaser deterministically
chooses one based on the precedence above.

During execution, the generated script picks the best asset for the current
OS/ARCH and installs the binary (extracting archives when needed).
For a given OS, assets with `goarch: all` can be selected regardless of the
runtime CPU architecture (for example, macOS universal artifacts).

## Publishing and pull requests

When publishing install scripts, GoReleaser does not run a preflight diff check
for `install_script` files. It always publishes the configured script files on
every run.

When `repository.pull_request.enabled` is true, GoReleaser opens a PR after
publishing through SCM APIs (GitHub/GitLab/Gitea). When pull requests are not enabled, it publishes
directly to the configured target branch/repository.
