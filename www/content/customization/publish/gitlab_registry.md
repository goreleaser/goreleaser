Publish your artifacts to a [GitLab generic package
registry](https://docs.gitlab.com/user/packages/generic_packages/).

## How it works

You can declare multiple GitLab registry instances.
All archives generated by your `archives` section will be pushed to each
configured registry.

The registry can belong to any project you can write to — including a project
other than the one being released. This allows you to [store all of your
packages in one GitLab
project](https://docs.gitlab.com/user/packages/workflows/project_registry/).

A minimal configuration looks like this:

```yaml {filename=".goreleaser.yaml"}
gitlab_registries:
  - name: production
    target: https://gitlab.com/api/v4/projects/<project-id>/packages/generic/{{ .ProjectName }}/{{ .Version }}/
```

### Target

The `target` is the GitLab API URL to upload the artifacts to (_without_ the
name of the artifact), in the format:

```
https://<gitlab-instance>/api/v4/projects/<project>/packages/generic/<package-name>/<package-version>/
```

`<project>` is either the project ID or the URL-encoded path of the project
(e.g. `mygroup%2Freleases`).

### Authentication

The token needs the `write_package_registry` scope (project or group [access
tokens](https://docs.gitlab.com/user/project/settings/project_access_tokens/),
[deploy tokens](https://docs.gitlab.com/user/project/deploy_tokens/), and
personal access tokens are all supported).

You can pass the token as a header using `custom_headers`:

```yaml {filename=".goreleaser.yaml"}
gitlab_registries:
  - name: production
    target: https://gitlab.com/api/v4/projects/<project-id>/packages/generic/{{ .ProjectName }}/{{ .Version }}/
    custom_headers:
      PRIVATE-TOKEN: "{{ .Env.REGISTRY_TOKEN }}"
```

Use the `DEPLOY-TOKEN` header for deploy tokens, or, when running in GitLab
CI/CD with a [job token](https://docs.gitlab.com/ci/jobs/ci_job_token/) that
has access to the target project:

```yaml {filename=".goreleaser.yaml"}
    custom_headers:
      JOB-TOKEN: "{{ .Env.CI_JOB_TOKEN }}"
```

Alternatively, you can use HTTP basic authentication by setting `username`
(or the environment variable `GITLAB_REGISTRY_<NAME>_USERNAME`) and either
`password` or the environment variable `GITLAB_REGISTRY_<NAME>_SECRET`, where
`<NAME>` is the uppercased `name` of the instance — e.g.
`GITLAB_REGISTRY_PRODUCTION_SECRET`. For deploy tokens, the username must be
the deploy token's username.

### Storing packages of multiple projects in one project

If you only want the artifacts in the designated registry project and not
attached to the release of the project being built, you can skip the release
upload:

```yaml {filename=".goreleaser.yaml"}
release:
  skip_upload: true

gitlab_registries:
  - name: releases
    target: https://git.company.com/api/v4/projects/mygroup%2Freleases/packages/generic/{{ .ProjectName }}/{{ .Version }}/
    custom_headers:
      PRIVATE-TOKEN: "{{ .Env.REGISTRY_TOKEN }}"
```

## Customization

Of course, you can customize a lot of things:

```yaml {filename=".goreleaser.yaml"}
gitlab_registries:
  # You can have multiple registry instances.
  - # Unique name of your registry instance. Used to identify the instance
    # and to build the name of the environment variable that holds the
    # secret.
    name: production

    # IDs of the artifacts you want to upload.
    ids:
      - foo
      - bar

    # File extensions to filter for.
    # This might be useful if you have multiple packages with different
    # extensions with the same ID, and need to upload each extension to
    # a different place (e.g. nFPM packages).
    exts:
      - deb
      - rpm

    # Matrix will run the upload for each possible combination of the given
    # values.
    # The keys will be available as template variables in the `target` and
    # `custom_headers` fields.
    #
    # This feature is only available in GoReleaser Pro.
    matrix:
      foo: [bar zaz]
      something: [foobar somethingelse anotherthing]

    # Upload mode. Valid options are `binary` and `archive`.
    #
    # If mode is `archive`, variables _Os_, _Arch_ and _Arm_ for target name
    #   are not supported. In that case these variables are empty.
    #
    # If mode is `binary`, you'll need to have the archives section setup with
    #   format "binary" as well.
    #
    # Default: 'archive'.
    mode: archive

    # URL of the generic package registry to deploy to.
    target: https://gitlab.com/api/v4/projects/<project-id>/packages/generic/{{ .ProjectName }}/{{ .Version }}/

    # Tells goreleaser not to append the artifact name to the target URL. You must do this manually
    custom_artifact_name: true

    # An optional username that will be used for the deployment for basic auth.
    #
    # Templates: allowed.
    username: deploytoken

    # An optional password that will be used for the deployment for basic auth.
    #
    # Templates: allowed.
    password: '{{ readFile "~/.config/foo" }}'

    # Client certificate and key (when provided, added as client cert to TLS connections)
    client_x509_cert: /path/to/client.cert.pem
    client_x509_key: /path/to/client.key.pem

    # A map of custom headers, e.g. to pass the authentication token.
    custom_headers:
      PRIVATE-TOKEN: "{{ .Env.REGISTRY_TOKEN }}"

    # Upload checksums.
    checksum: true

    # Upload metadata.json and artifacts.json.
    meta: true

    # Upload signatures.
    signature: true

    # Skip this upload configuration.
    #
    # Templates: allowed.
    skip: "{{gt .Patch 0}}"

    # Certificate chain used to validate server certificates
    trusted_certificates: |
      -----BEGIN CERTIFICATE-----
      MIIDrjCCApagAwIBAgIIShr2zchZo+8wDQYJKoZIhvcNAQENBQAwNTEXMBUGA1UE
      ...(edited content)...
      TyzMJasj5BPZrmKjJb6O/tOtEIJ66xPSBTxPShkEYHnB7A==
      -----END CERTIFICATE-----

    # You can add extra pre-existing files to the upload.
    #
    # The filename on the release will be the last part of the path (base).
    # If another file with the same name exists, the last one found will be used.
    # These globs can also include templates.
    extra_files:
      - glob: ./path/to/file.txt
      - glob: ./glob/**/to/**/file/**/*
      - glob: ./glob/foo/to/bar/file/foobar/override_from_previous
      - glob: ./single_file.txt
        # Templates: allowed.
        name_template: file.txt # note that this only works if glob matches 1 file only

    # Additional templated extra files to uploaded.
    # Those files will have their contents pass through the template engine,
    # and its results will be uploaded.
    #
    # {{< g_inline_pro >}}
    # Templates: allowed.
    templated_extra_files:
      - src: LICENSE.tpl
        dst: LICENSE.txt

    # Upload only the files defined in extra_files.
    extra_files_only: true
```
